# MailSender

Десктопное приложение (Windows, Python) для **B2B-аутрича по-белому**
и **рассылки по своей базе** через корпоративный SMTP.

Интерфейс — в стиле «матовое стекло» (glassmorphism): тёмная тема, фото-фон,
воздушные скруглённые карточки, канбан-доска контактов и плавающая кнопка
действия. Рендерится во встроенном окне через `pywebview` (на Windows —
движок WebView2/Edge). Весь Python-бэкенд общий с классической Qt-версией.

Инструмент рассчитан на работу по закону: он отправляет письма контактам,
которые уже есть в вашей базе (дали согласие, оставили контакт, обменялись
визитками), и не занимается сбором чужих адресов с сайтов.

## Возможности

- **Импорт своей базы** из CSV / TSV / TXT / XLSX с сопоставлением колонок,
  валидацией и удалением дублей.
- **Отправка через корпоративный SMTP** (STARTTLS 587 или SSL 465), пароль
  хранится в системном хранилище (Windows Credential Manager через `keyring`),
  а не в файле настроек.
- **Персонализация** письма: `{{first_name}}`, `{{last_name}}`, `{{company}}`,
  `{{email}}` и любые свои поля; дефолт через `{{first_name|коллеги}}`.
- **Темп и лимиты**: паузы между письмами, пачки, лимиты в час/сутки —
  чтобы не попасть в блокировки и чёрные списки.
- **Модель 1-to-1 аутрича**: каждому уходит одно персональное письмо с рабочего
  ящика, как обычное деловое, — без блоков «отписаться» и признаков массовой
  рассылки. Подпись (организация/адрес) добавляется, только если заполнена.
- **Стоп-лист**: адреса из стоп-листа никогда не получают писем — это механизм
  отказа: если получатель просит не писать, менеджер вносит его сюда.
- **Чтение ответов по IMAP** и автоматический сбор писем с отказом («не писать»,
  «stop») в стоп-лист.
- **Полный лог доставки** в SQLite; повторный запуск кампании не шлёт письма
  тем, кому уже успешно отправлено.

## Установка

```bash
pip install -r requirements.txt
```

Требуется Python 3.10+ (проверено на 3.14).

## Запуск

Три режима из одного кода:

```bash
python run.py          # десктоп: стеклянное окно (pywebview)
python run.py web      # веб-сервер (FastAPI) на http://localhost:8000
python run.py qt       # классический интерфейс на Qt
```

## Веб-версия на VPS (Docker)

Для отдела продаж: один общий инстанс на сервере, вход по IP.

```bash
cp .env.example .env
# отредактируйте .env: задайте MAILSENDER_ACCESS_PASSWORD
docker compose up -d --build
```

Панель откроется на `http://<IP-сервера>:8000`. Данные (БД, настройки,
пароль SMTP) хранятся в томе `./data` и переживают пересборку контейнера.

**Безопасность — обязательно к прочтению.** Панель шлёт письма через ваш
SMTP, поэтому доступ к ней закрыт **общим паролем** (`MAILSENDER_ACCESS_PASSWORD`
из `.env`) — это один ключ на всю команду, а не регистрация. Без пароля панель
на публичном IP стала бы открытым спам-релеем. Рекомендуется также:

- закрыть порт файрволом на нужные IP или пустить через reverse-proxy;
- поставить перед сервисом HTTPS (nginx/Caddy + Let's Encrypt), т.к. пароль
  и данные идут по сети;
- пароль SMTP можно не вводить в UI, а передать секретом `MAILSENDER_SMTP_PASSWORD`.

Переменные окружения:

| Переменная | Назначение |
|---|---|
| `MAILSENDER_ACCESS_PASSWORD` | Общий пароль на вход (обязательно) |
| `MAILSENDER_SMTP_PASSWORD` | Пароль SMTP секретом (необязательно) |
| `MAILSENDER_DATA_DIR` | Каталог данных внутри контейнера (по умолч. `/data`) |
| `MAILSENDER_HOST` / `MAILSENDER_PORT` | Адрес/порт при `python run.py web` |

## Порядок работы

1. **Настройки** — заполните SMTP (хост, порт, логин, пароль), отправителя
   (имя, email, при желании — организация/адрес для подписи) и лимиты.
   Нажмите «Проверить SMTP».
2. **Контакты** — импортируйте свою базу из файла и сопоставьте колонки.
3. **Письмо и рассылка** — напишите тему и текст с плейсхолдерами, посмотрите
   превью на реальном контакте, отправьте себе тестовое письмо, затем запустите
   рассылку. Прогресс и лог видны внизу.
4. **Ответы** — по IMAP подтяните ответы адресатов; письма с отказом
   автоматически собираются в стоп-лист.

## Как работать аккуратно

Модель — персональный B2B-аутрич: одно письмо каждому контакту с рабочего ящика,
дальше менеджер общается лично. Чтобы это не превращалось в спам и письма
доходили:

- пишите только релевантным компаниям/людям, персонально;
- один контакт — одно первое письмо (повторно программа тем же не шлёт);
- если человек просит не писать — вносите его в стоп-лист (или соберите отказы
  из ответов по IMAP), и он больше не получит писем;
- держите разумный темп (лимиты в настройках), чтобы не ловить блокировки.

## Структура

```
mailsender/
  config.py        # настройки + пароль в keyring
  storage.py       # SQLite: контакты, кампании, лог, стоп-лист
  contacts.py      # импорт и валидация адресов
  templates.py     # персонализация + обязательный подвал
  smtp_client.py   # отправка
  imap_client.py   # чтение ответов, сбор отписок
  campaign.py      # движок рассылки (throttling, лимиты, лог)
  api.py           # общий Api (используют и desktop, и web)
  webmain.py       # точка входа desktop-версии (pywebview)
  webserver.py     # веб-сервер FastAPI: /api, /events (SSE), вход, статика
  webapp/          # HTML/CSS/JS стеклянного интерфейса
    index.html
    style.css      # дизайн-система glassmorphism
    app.js         # работает и в pywebview, и по HTTP
    background.svg # фон-сцена (оригинальная графика)
  ui/              # классический интерфейс на PySide6
  main.py          # точка входа Qt-версии
run.py             # запуск: (пусто)=desktop, web=сервер, qt=Qt
Dockerfile             # образ веб-сервера
docker-compose.yml     # запуск на VPS с томом данных
requirements-web.txt   # зависимости контейнера (без Qt/pywebview)
```

Один фронтенд обслуживает оба режима: в pywebview он ходит через JS-мост,
в браузере — по HTTP (`/api/<метод>`), а прогресс рассылки получает через SSE.

Веб-интерфейс можно открыть и в обычном браузере для правки дизайна — при
отсутствии Python-бэкенда включается демо-режим с примерными данными
(`python -m http.server` в папке `webapp`).
